## 1. Hash chain definition

Per-applicant chain (v1). The next entry's `entryHash` is:

```
SHA-256(
  sequence       (8B, big-endian uint64)
  || prevHash    (32B raw bytes)
  || canonicalJson(payload)  (UTF-8 bytes)
  || createdAt   (UTF-8 of ISO-8601 with millis + Z)
)
```

- `sequence` starts at 1 per applicant, monotonically increases.
- `prevHash` for `sequence=1` is `GENESIS_HASH` (32 zero bytes).
- `createdAt` is generated by the service at `stamp()` time, **not** by the caller.
- `canonicalJson` is sorted-keys-at-every-depth, no whitespace, JS-native numbers.
  We intentionally don't implement full RFC 8785 — see hashing.ts comment.

## 2. Database schema (Lane A — for reference)

```sql
CREATE TABLE compliance_tape (
  id            uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  applicant_id  uuid REFERENCES applicants(id) ON DELETE RESTRICT,
  sequence      bigint NOT NULL,
  kind          text NOT NULL,
  citation      text NOT NULL,
  payload       jsonb NOT NULL,
  prev_hash     bytea NOT NULL,
  entry_hash    bytea NOT NULL,
  session_id    text,
  created_at    timestamptz NOT NULL DEFAULT now(),
  UNIQUE (applicant_id, sequence),
  UNIQUE (kind, session_id)  -- idempotency, NULL session_id is ignored by PG
);

CREATE INDEX compliance_tape_applicant_seq ON compliance_tape (applicant_id, sequence);

-- Append-only enforcement
CREATE OR REPLACE FUNCTION compliance_tape_reject_mutation()
RETURNS trigger AS $$
BEGIN RAISE EXCEPTION 'compliance_tape is append-only'; END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER compliance_tape_no_update BEFORE UPDATE ON compliance_tape
  FOR EACH ROW EXECUTE FUNCTION compliance_tape_reject_mutation();
CREATE TRIGGER compliance_tape_no_delete BEFORE DELETE ON compliance_tape
  FOR EACH ROW EXECUTE FUNCTION compliance_tape_reject_mutation();
```

Service code converts buffers to/from hex when crossing the JS boundary
(`hashToHex` / `hashFromHex`). Postgres stores raw `bytea`.

## 3. Service API (Lane B)

```ts
interface TapeService {
  stamp(event: TapeEvent, scope: TapeScope): Promise<TapeEntry>;
  verify(scope: TapeScope): Promise<VerifyResult>;
  exportPdf(scope: TapeScope): Promise<Buffer>;
}
```

- `stamp()` must run inside a transaction: read tail, compute hash, insert.
  If two stamps race for the same `(applicant_id, sequence)` the UNIQUE
  constraint rejects the loser; service retries up to 3x.
- `stamp()` is idempotent on `(kind, sessionId)` — preserves current stub
  behavior. Returns the existing row on dup.
- `verify()` re-runs `computeEntryHash` over every row in order and returns
  `{ok, lastSequence, brokeAt?, reason?}`.
- `exportPdf()` returns a `Buffer`. Footer of every page shows the rolling
  SHA-256 of all entries so far. v1 paginates at 50 entries per page.

## 4. HTTP API (Lane D)

All routes operator-only (`hasMinRole('operator')`). Mounted at
`/api/compliance-tape`. See `api-contract.ts` for exhaustive shapes.

| Method | Path | Notes |
|---|---|---|
| GET | `/api/compliance-tape?applicantId=...` | List, paginated by `afterSequence` + `limit` |
| GET | `/api/compliance-tape/verify?applicantId=...` | Returns `VerifyResult` |
| GET | `/api/compliance-tape/export.pdf?applicantId=...` | `application/pdf` binary |

v1 requires `applicantId`. Passing `scope=global` returns
`501 {code:"global_scope_not_implemented"}`.

## 5. JSON-LD payloads (Lane C)

Every kind has a `make<EventName>Payload(args)` in `src/modules/tape/events/<kind>.ts`
returning a `TapeJsonLdPayload`. Required fields on every payload:

```ts
{
  "@context": "https://frank-pilot.example/compliance-tape/v1",   // stub URL; see §7
  "@type":    "ComplianceEvent.<EventName>",
  actorId,         // string | null
  subjectId,       // string | null — applicant id for applicant-scoped events
  ruleCitation,    // pulled from TAPE_CITATIONS[kind]
  evidence: { … }, // kind-specific fields; document each in events/<kind>.ts
}
```

Don't fabricate `actorId` — pass `null` for system events.

## 6. Feature flag + cutover (Lane G)

`COMPLIANCE_TAPE_V2_ENABLED=true|false` (env). Lane G's port:

```ts
await stampTape({ kind, actor, payload, sessionId });        // legacy NDJSON
if (process.env.COMPLIANCE_TAPE_V2_ENABLED === "true") {
  await tape.stamp({ kind, payload: jsonld, sessionId }, scope);  // new path
}
```

Dual-write stays until one prod deploy is clean and Lane F's verify matches
NDJSON line-count. Then a follow-up PR drops the legacy writer.

## 7. Known stubs / v1 limitations

- `@context` URL points at `frank-pilot.example` until we own a public schema.
- Global-scope reads/exports return 501.
- PDF generation is in-process (no worker queue). Fine for demo scale.
- Hash includes `createdAt` so timezone bugs surface as verify failures —
  intentional.
- No tenant-facing view in v1; operator-only.

## 8. Don't-do list for lane owners

- Don't change canonicalJson, the digest input order, or the genesis hash.
- Don't add a new `TapeStampKind` without also adding its citation to
  `TAPE_CITATIONS` and a maker under `events/`.
- Don't write to `compliance_tape` outside `TapeRepository.insert`.
- Don't edit `src/modules/applicants/routes.ts` or `src/modules/auth/routes.ts`
  unless you are Lane G.
- Don't drop the legacy NDJSON writer in this packet — that's a follow-up.
